# Phase 4 — 실시간 + 백그라운드 (슬라이스 2개를 통짜로)

기술(WebSocket, BullMQ)이 아니라 **기능 슬라이스 단위로 진행**. 각 슬라이스를 DB→API→UI까지 관통시킨 뒤 다음 슬라이스로. Phase 3에서 완성한 servers 슬라이스의 관통 경로를 복제하는 연습.

## Slice A — `features/metrics` (실시간 메트릭)

끝까지 한 번에: 테이블 → 수집 API → 스트림 → 차트

- Drizzle 테이블 + 마이그레이션 (이 슬라이스 소유)
- ingest 엔드포인트: 가짜 에이전트 스크립트가 주기적으로 POST
- WebSocket 스트리밍 (org 스코프 인증 포함 — 소켓 인증이 학습 포인트)
- 프론트 `features/metrics`: 실시간 차트
- 오래된 메트릭 집계·정리 반복 잡 (BullMQ 첫 도입)

**완료 기준**: 에이전트 켜면 차트가 실시간으로 움직임. servers 슬라이스 코드 수정이 (등록 화면에 링크 추가 수준 이상으로) 필요했다면 경계 설계 재점검

## Slice B — `features/alerts` (임계치 알림)

- 알림 규칙 테이블 (서버별 임계치)
- BullMQ: 임계치 초과 시 알림 잡 생성 (발송은 콘솔/웹훅 stub)
- 알림 이력 API + 프론트 `features/alerts`: 규칙 설정 + 알림 목록 UI

**완료 기준**: 임계치 넘기면 잡 처리 로그 + UI 알림 목록에 표시. metrics 슬라이스와의 연결점(임계치 평가가 메트릭을 읽는 지점)을 어디에 뒀는지 한 문단으로 정리

## Cross-cutting — Redis 캐싱

- 대시보드 요약 쿼리 캐시 + 무효화 전략 (슬라이스가 아니라 infra 관심사임을 확인)

**완료 기준**: 캐시 히트/미스를 로그로 검증

## 구현 전 던져야 할 질문

1. WebSocket vs SSE vs long polling — 이 앱(서버→클라 단방향 스트림)엔 뭐가 최소인가? WebSocket을 골랐다면 왜?
2. WebSocket 핸드셰이크에서 세션 인증은 어떻게 하나? 연결 후 세션이 만료되면?
3. 클라이언트가 느릴 때(backpressure) 메시지는 어떻게 되나? 버퍼가 터지면?
4. BullMQ 잡이 실패하면? 재시도 중 중복 실행되면? — idempotent 잡 설계란?
5. "임계치 초과 → 알림"에서 같은 알림이 초당 수십 개 생기는 것(알림 폭풍)을 어떻게 막나? (debounce, 상태 전이 기반 알림)
6. 캐시 무효화: TTL만 쓸 때와 쓰기 시점 무효화의 트레이드오프는? 이 대시보드엔 뭐가 맞나?
7. 시계열 메트릭을 일반 테이블에 계속 쌓으면 언제 무너지나? 집계(rollup) + 원본 삭제 전략은?
8. alerts 슬라이스가 metrics 데이터를 필요로 할 때 — repository 직접 import, service 경유, 이벤트 발행 중 뭘 택하나? 각각 결합도가 어떻게 다른가?
9. 왜 캐싱은 슬라이스가 아니라 cross-cutting인가? 슬라이스로 만들면 뭐가 이상해지나?
