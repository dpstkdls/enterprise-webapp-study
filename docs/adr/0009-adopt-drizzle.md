# ADR-0009: ORM — Drizzle 채택

- 상태: 승인됨
- 날짜: 2026-07-21
- 관련: [apps/api/src/infra/db.ts](../../apps/api/src/infra/db.ts), [apps/api/drizzle.config.ts](../../apps/api/drizzle.config.ts), 이슈 #34 #36

## 배경

DB 접근 도구가 필요하다. 요구사항: TS 타입 안전, 스키마→마이그레이션
파이프라인, 그리고 vertical slice 구조(ADR-0008)와의 궁합 — 테이블 정의를
슬라이스 폴더에 둘 수 있어야 한다. 이 레포는 학습 목적이라 "도구가 SQL을
얼마나 숨기는가"도 기준이다 — 숨길수록 편하지만 배우는 게 없다.

## 검토한 대안

### A. Prisma

- 장점: 가장 성숙한 생태계, 마이그레이션/스튜디오 등 도구 완성도 최고
- 기각 이유: 스키마가 자체 DSL(`schema.prisma`) 단일 파일 — 테이블 정의를
  슬라이스별로 흩어 둘 수 없어 ADR-0008과 정면 충돌. 쿼리 API가 SQL에서
  가장 멀어 생성되는 쿼리 예측이 어렵고, 생성 클라이언트/엔진 등 런타임이
  무겁다. 학습 목적에서 "SQL이 안 보이는 것"은 단점

### B. Kysely

- 장점: 타입 안전 쿼리 빌더로는 최상. SQL 1:1이라 학습 목적에 가장 부합
- 기각 이유: 쿼리 빌더*만* 있다 — 스키마 정의/마이그레이션 생성이 없어
  kysely-codegen + 별도 마이그레이터를 조합해야 함. "TS로 테이블 정의 →
  SQL 마이그레이션 자동 생성" 파이프라인을 원하면 결국 직접 조립

### C. Drizzle ← 채택

- 테이블 정의가 평범한 TS 코드 — 슬라이스 폴더에 두고 drizzle-kit이
  `features/**/*.schema.ts` glob으로 수집. ADR-0008과 맞물림
- `drizzle-kit generate`가 스키마 diff에서 SQL 마이그레이션 파일 생성 →
  파일 커밋 → `migrate` 적용. SQL이 눈에 보이는 채로 파이프라인 자동화
- 쿼리 API가 SQL 1:1 (`select().from().where()`) — 예측 가능, 학습에 적합
- 코드 생성 없이 타입 추론만으로 동작 — 런타임 경량

## 반론과 대응

- **"0.x 버전, Prisma 대비 성숙도"** — 사실. 단 스키마·쿼리 API는 안정권이고,
  이 레포 규모의 사용 범위(테이블 정의, CRUD, 마이그레이션)에서 미성숙을
  체감한 지점은 아직 없음. 문제가 생기면 쿼리 빌더 계층은 Kysely로 갈아탈
  수 있는 구조(repository 파일에 격리됨)
- **"함정이 있다"** — 실제로 밟았다: `casing: "snake_case"`는 drizzle 클라이언트
  생성 지점마다 지정해야 하며, 빠뜨리면 컬럼 매핑이 조용히 어긋난다.
  integration 테스트(#38)가 이 클래스의 실수를 잡는 안전망

## 결정

ORM은 Drizzle. 테이블 정의는 각 슬라이스의 `*.schema.ts`가 소유하고,
마이그레이션은 `apps/api/drizzle/`에 전역 하나로 관리한다.

## 결과

- 테이블 정의는 슬라이스별로 흩어져도 마이그레이션이 전역 하나인 이유:
  DB 스키마는 전역 상태다 — 적용 순서가 하나의 이력이어야 하고, FK 등
  슬라이스 경계를 넘는 제약이 존재할 수 있다
- 마이그레이션 파일은 손으로 고치지 않는다 — generate 산출물이며, 수정이
  필요하면 스키마(TS)를 고치고 재생성. 적용된 마이그레이션은 불변
- 클라이언트 생성 규칙: `drizzle(pool, { casing: "snake_case" })` —
  앱(infra/db.ts)과 테스트가 동일 설정을 써야 함
- 재검토 조건: 복잡한 조인/집계에서 Drizzle 쿼리 API가 걸림돌이 되면
  해당 repository만 raw SQL(`sql` 태그) 또는 Kysely로 대체 검토
