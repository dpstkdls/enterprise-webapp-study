# ADR-0008: api 코드 구조 — 레이어드 대신 vertical slice

- 상태: 승인됨
- 날짜: 2026-07-21
- 관련: [apps/api/src](../../apps/api/src), [docs/phase-1.md](../phase-1.md), 이슈 #31, PR #41

## 배경

Phase 1부터 api에 기능이 붙기 시작한다 — servers를 시작으로 이후 페이즈의
auth/metrics/alerts까지. 기능이 늘어날 때 코드가 어떻게 배치되어야
"기능 하나 추가/수정"의 변경 범위가 좁게 유지되는지가 이 결정의 핵심이다.

## 검토한 대안

### A. 레이어드 (`routes/`, `services/`, `repositories/`)

- 장점: 레이어 규율이 폴더 구조로 보임. 튜토리얼 대부분이 이 구조라 익숙함
- 기각 이유: 응집의 축이 틀림. 실제 변경 단위는 "기능"인데 폴더는 "역할"로
  묶여 있어, 기능 하나를 수정하면 3개 폴더를 순회하고 기능 하나를 추가하면
  모든 레이어에 파일이 흩어짐. 기능이 늘수록 각 폴더가 서로 무관한 파일들의
  더미가 됨

### B. 슬라이스당 파일 하나 (`features/servers.ts`)

- 장점: 최소 파일 수, 작은 기능엔 충분
- 기각 이유: 파일 안에서도 역할 경계는 필요하다 — 테스트가 증명.
  service unit 테스트는 repository를 모듈째 모킹하는데(#38), 같은 파일에
  있으면 대체 경계가 없음. zod 스키마/도메인 로직/SQL이 한 파일에 섞이면
  결국 주석으로 구역을 나누게 됨 — 그럴 바엔 파일이 낫다

### C. vertical slice — 기능별 폴더, 폴더 안에서 역할별 파일 ← 채택

```
features/servers/
├── servers.routes.ts      # 라우트 + zod 스키마 (HTTP 경계)
├── servers.service.ts     # 도메인 로직 (not found 등 도메인 에러)
├── servers.repository.ts  # DB 접근
├── servers.schema.ts      # Drizzle 테이블 정의
└── *.test.ts
```

- 한 기능의 라우트→검증→로직→DB가 폴더 하나에 응집. 새 기능 = 새 폴더 +
  app.ts 등록 1줄
- 슬라이스는 Fastify plugin으로 조립 — DI 컨테이너 없이 `decorate`(db 등)로
  의존성 전달
- 테이블 정의까지 슬라이스가 소유 — drizzle-kit이 `features/**/*.schema.ts`
  glob으로 수집 (ADR-0009)

## 반론과 대응

- **"레이어 규율이 안 보인다"** — 슬라이스 안의 파일 명명(routes/service/
  repository)이 같은 규율을 폴더 대신 파일 수준에서 유지한다. import 방향
  (routes→service→repository)이 규칙이고, 위반은 리뷰에서 잡는다
- **"슬라이스 간 코드 중복"** — 의도된 비용. 슬라이스 간 성급한 공유가
  만드는 결합보다 약간의 중복이 싸다. 공유는 실제로 2개 이상 슬라이스가
  같은 코드를 필요로 하는 시점에만 승격한다
- **"여러 슬라이스를 관통하는 기능(트랜잭션, 리포트)"** — 그 기능 자체를
  슬라이스로 만들고 다른 슬라이스의 service를 호출한다. 그래도 안 맞으면
  그때 경계를 재조정 — 지금 규모에서 미리 설계하지 않는다

## 결정

api는 `infra/` + `features/` 구조. 기능은 vertical slice(기능별 폴더)로,
슬라이스 안은 역할별 파일로 나눈다. 슬라이스 등록은 app.ts 한 곳.

## 결과

- #37에서 실증: servers CRUD 추가가 `features/servers/` + app.ts 1줄로 완결
- `infra/` 승격 기준 — "2개 이상 슬라이스가 실제로 사용"할 때만. 도메인과
  무관한 기술 코드(db, logger, 에러 핸들러, config)만 허용. 도메인 코드가
  infra로 가려 하면 별도 슬라이스로 만드는 게 먼저 (쓰레기장 방지)
- 셀프 점검 기준 유지: "가짜 기능 하나를 추가한다면 features/ 폴더 하나로
  끝나는가?" — infra를 고치게 되면 경계 재조정 신호
- 재검토 조건: 슬라이스 수가 늘어 슬라이스 간 호출 그래프가 복잡해지면
  (순환 참조 등장 등) 모듈 경계/공개 인터페이스 규칙 도입 검토
