# ADR-0005: Fastify API + React SPA (Next.js 미채택)

- 상태: 승인됨
- 날짜: 2026-07-15 (기록일 — 결정 자체는 Phase 0 초기)
- 관련: [apps/api](../../apps/api), [apps/web](../../apps/web)

## 배경

풀스택 웹앱의 프레임워크 구성을 정해야 한다. 이 프로젝트의 목적은
취업용 포트폴리오 + 백엔드/인프라 역량 학습 — 프레임워크가 감춰주는 층을
직접 만져보는 것이 핵심 가치다.

## 검토한 대안

### A. Next.js 풀스택 (API Routes / Server Actions)

- 장점: 단일 프레임워크, SSR/RSC, 배포 생태계(Vercel) 성숙
- 단점: 서버 층이 프레임워크에 봉인됨 — 미들웨어, 라이프사이클, 프로세스 관리,
  graceful shutdown 같은 백엔드 학습 포인트가 추상화 뒤로 사라짐.
  학습 목적과 정면 충돌

### B. Express + SPA

- 장점: 자료 최다, 업계 레거시 표준
- 단점: 유지보수 동력 약함, 타입 지원 빈약, async 에러 처리 수동.
  2026년 신규 프로젝트 선택지로는 관성 외 근거 없음

### C. Fastify + React SPA(Vite) ← 채택

- Fastify: 타입 우선 설계, 플러그인 캡슐화 모델, 스키마 기반 검증/직렬화,
  구조화 로깅(pino) 내장 — "제대로 된 백엔드"의 학습 포인트가 전부 노출됨
- React+Vite SPA: 프론트는 표준 구성으로 두고 학습 무게를 백엔드/인프라에 배분
- 단점: SSR/SEO 없음(내부 도구 성격이라 무관), 서버-클라이언트 통합을
  직접 설계해야 함(그게 목적)

## 결정

Fastify API + React SPA(Vite), 별도 프로세스로 분리. 통신은 HTTP API,
계약(타입)은 `@ews/shared`로 공유.

## 결과

- 백엔드 관심사(검증, 로깅, 에러 처리, 헬스체크, graceful shutdown)를
  전부 직접 구현 — Phase 1 이후의 학습 재료
- SPA 배포와 API 배포가 분리됨 — 인프라 학습(리버스 프록시, 컨테이너)이
  자연스럽게 따라옴
- SSR이 필요해지는 요구가 생기면 그때 별도 ADR로 재검토
